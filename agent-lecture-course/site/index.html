Presenter Notes

总览备注

这一页不要讲太久，作用是把全场视线先拉到主命题上：今天我们不是在看又一个聊天框，而是在看 AI 为什么开始像“系统”而不是“答题器”。

第1讲备注

这一讲要先破除误解，再上概念。先让大家觉得“好像确实不是一个东西”，再解释为什么不是。重点句：Agent 不是更会说，而是更会沿着过程行动。

第2讲备注

避免把现场带成技术排障大会。重点不是每个人都部署到完美，而是让大家看懂配置项到底在控制什么，以及为什么权限、Skill、渠道会改变它的能力边界。

第3讲备注

这一讲核心不是列举工具，而是训练判断。反复提醒：可以把重复劳动交给 Agent，但不要把理解、判断和表达外包出去。

结语备注

结尾不要变成“谢谢观看”。这一页更像收剑入鞘：Agent 很强，但真正决定你会不会被带偏的，是你的判断力，而不是它的参数表。

Lecture 01

Agent 是什么，为什么它不只是 ChatGPT

建议时长：120 分钟 目标：打破“更强聊天框”误解

Opening Prompt

如果你说：“帮我整理一篇英语文章，再生成复习提纲。”

Chat AI

像嘴很快的顾问

擅长解释、总结、回答问题，但通常停留在“说”的层面。你问一步，它答一步，更多是给建议，不一定真的去动手。

  • 一问一答
  • 偏回答，不偏执行
  • 通常不接管完整流程
Agent

像能动手的助理

擅长围绕目标拆步骤、调工具、跑流程、回报结果。它不只是“会说得更长”，而是“会沿着过程行动”。

  • 目标驱动
  • 会规划、会调工具
  • 更接近完整任务闭环

被动响应

聊天 AI

像你每次都得叫它一下，它才抬头看你。

目标驱动

Agent

你给它目标，它开始琢磨要分几步走。

封闭功能

聊天 AI

像一个坐在桌前很会讲道理的人。

Skill 扩展

Agent

像一个能不断学新技能、带新装备的执行系统。

Why Now

为什么 OpenClaw 一类工具会在这一轮突然变热？

时机到了

AI 发展刚好卡在一个临界点：大家已经厌倦“只会答”的工具，开始渴望“真的能干活”的系统。

画面感变强了

过去很多 Agent 讨论停留在概念里，这一轮大家第一次更直观地看到“目标输入后，系统真的去跑流程”。

本地优先很抓人

“文件不一定都要上传云端”“能在自己机器上跑”这件事，对隐私敏感和重度工作流用户都很有吸引力。

社交传播太强

一旦出现“像科幻起飞”的案例，传播速度会非常快。人们不一定先懂原理，但会先被场景震住。

Advantage Matrix

OpenClaw 一类 Agent 的 4 个差异化优势

本地闭环

不是每一步都要把完整文件上传出去，很多动作可以在本地环境完成。

常驻待命

不像传统 AI 用完即走，它更像后台驻留的值班员，可以接定时和触发式任务。

Skill 扩展

能力不是固定死的，可以装技能包，像给系统追加新的模块和手艺。

跨平台接入

不必困在单一网页里，可以接入飞书、群聊、企业办公平台，嵌进原有工作流。

Reality Check

也别神化它：它不是万能数字分身

  • 它适合规则明确、步骤标准化的任务
  • 它不适合需要连续决策、频繁变更目标的开放任务
  • 它看起来像“全自动”，但背后仍然需要人定义边界
  • 它不是终极形态，更像是从聊天 AI 走向执行系统的过渡阶段

Core Structure

一个 Agent 通常由什么组成？

大脑

模型层，负责理解任务、推理路线、安排步骤。大脑再聪明，没有手脚也只能当军师。

手脚

工具层，负责读文件、开网页、执行动作。真正让 Agent 和普通聊天机器人拉开差距的，往往就是这双“手脚”。

记忆

像本地备忘录，帮助它保留上下文、偏好和历史重点，不至于每一轮都像失忆了一样重新开始。

渠道

决定你是通过网页、飞书、群聊还是别的平台叫它做事。门开在哪，决定你和它平时怎么见面。

钥匙

权限决定它能进哪些房间。钥匙太少，它什么也做不了；钥匙太多，它可能一激动把锅端走。

Execution Chain

Agent 的典型工作流程

先告诉它目标，而不是把每一步都替它念出来。
“我们不是在讨论谁更聪明，我们在讨论谁除了会说，还多长出了手和脚。”

Lecture 02

把 Agent 跑起来，理解部署与配置

建议时长：120 分钟 目标：跑起来，也看懂为什么能跑
Path A

本地部署

自由度高，隐私更强，但第一次上手像自己装修房子，爽是爽，就是容易先和水电打架。

Path B

平台托管

更省心，适合课堂展示和快速体验，像拎包入住的精装公寓，但平台给你什么结构，你就得先顺着住。

Path C

一键部署

最快上手，最适合第一口尝鲜，但别把“点一下就能跑”误会成“没有代价”。

Platform Atlas

现实里常见的 4 种落地形态

原生 OpenClaw

最能体现 Agent 原味：本地优先、系统权限、可深度定制，但也最考验配置耐心。

  • 自由度高
  • 门槛高
  • 适合深度折腾

Kimi 托管型

像平台给你开了一个云端沙箱，体验非常丝滑，适合先感受“Agent 跑起来”的感觉。

  • 上手快
  • 适合演示
  • 受平台环境约束

扣子场景型

更偏平台化产品和应用编排，适合快速接技能、接场景、接内容生产工作流。

  • 对小白友好
  • 技能安装直观
  • 商业化约束更明显

飞书集成型

最大的价值是融入已有办公流：消息、待办、文档、多维表格这些能力都能串起来。

  • 工作流顺滑
  • 适合团队/课堂演示
  • 配置机器人和权限是关键

Classroom Recommendation

课堂主方案：轻门槛部署，重理解配置

课堂里最重要的不是“最硬核”，而是“能跑起来、能看懂、能讲明白”。所以推荐优先演示低门槛方案， 再补充原生 OpenClaw、Kimi、扣子、飞书等不同类型的现实位置。

Official Quickstart

按官方文档，从零到第一个可用聊天的最短路径

Step 1

先确认环境：官方入门页要求 Node 22 或更高版本。

Step 2

安装 OpenClaw，然后运行 `openclaw onboard --install-daemon` 或 `openclaw onboard` 完成向导。

Step 3

检查 Gateway 是否正常：`openclaw gateway status`。

Step 4

最快体验不是先配渠道，而是先开控制台：`openclaw dashboard`，直接在浏览器聊天。

这条线很适合课堂演示，因为它先把“能用”跑起来，再决定要不要接聊天渠道。

Configuration Console

配置项背后的意思

模型

决定它怎么思考，像给大脑装发动机。发动机不是越响越好，关键看适不适合当前任务。

工具

决定它能动哪些手脚。没有工具，它顶多是个会说话的军师。

Skill

像可插拔能力包。不是你每次手动点 App，而是装上之后，它在合适的时候可以自己调用。

权限

像钥匙串。钥匙太少，它站在门外叹气；钥匙太多，它可能热心过头。

记忆

帮助它维持上下文连续，不至于每一轮都像第一次见面。

渠道

决定你从哪里叫它：网页、飞书、群聊还是别的平台。门开在哪，工作流就延伸到哪。

Model Setup

模型怎么配，官方推荐怎么做？

  • 先通过 `openclaw onboard` 或 `openclaw configure` 完成 provider 认证
  • 再用 `openclaw models status` 查看当前默认模型和认证状态
  • 用 `openclaw models list` 看可用模型
  • 用 `openclaw models set provider/model` 设置默认模型
  • 如果要写到配置里，可以在 `agents.defaults.model.primary` 指定如 `anthropic/...` 或 `openai/...`

官方文档强调的是“先选 provider，再设默认模型”，而不是一上来先改一堆配置文件。

Model Strategy

课堂里怎么讲“模型选择”才不空？

强模型

更适合复杂规划、长任务和高质量输出，但成本更高。

轻模型

更适合快速、便宜、日常型任务，但容易在复杂任务上掉链子。

课堂建议

演示时用稳定模型，避免现场把时间消耗在模型抽风和额度问题上。

思维提醒

模型决定“脑子”，但 Agent 的体验好不好，往往同样取决于工具、权限和渠道。

Skill Lifecycle

一个 Skill 从“听说过”到“真的可用”要经过什么？

发现

从 ClawHub、GitHub、课程资料、社区案例里找到合适能力。

安装

直接让 Agent 装，或者把技能包丢给它，让它自己接入。

配置

很多技能不只是“装完就行”，还需要补 API、Token、URL、权限。

编排

真正有价值的不是单技能，而是多个技能被任务编排成一个完整闭环。

Channel Integration

接入飞书这类平台时，真正要做的是什么？

  • 创建机器人或应用
  • 开通消息、文件、资源等必要权限
  • 配置事件接收或长连接
  • 把 App ID / Secret 等关键参数交给 Agent 或平台配置页
  • 完成配对后，再验证它能不能在真实聊天场景里闭环

这也是为什么“能接入渠道”很重要，因为 Agent 真正强的地方往往不是网页里，而是融进你原本的工作流里。

Feishu Official Path

按官方文档接入飞书，主线其实很清楚

  1. 先在飞书开放平台创建企业应用，拿到 App ID 和 App Secret。
  2. 然后运行 `openclaw channels add`，选择 Feishu 并填入凭证。
  3. 如果使用 WebSocket/长连接模式，不必暴露公网 webhook。
  4. 配置完成后，用 `openclaw gateway status`、`openclaw gateway restart`、`openclaw logs --follow` 检查状态。
  5. 如果要手动写配置，可以在 `~/.openclaw/openclaw.json` 里配置 `channels.feishu.accounts.main`。

这一块很适合现场强调：渠道接入本质上是在给 Agent 装一张“嘴”和“一对耳朵”。

WeChat Path

微信可以怎么接？要分清“官方主线”和“实战方案”

  • 我查到的 OpenClaw 官方文档主线里，飞书路径最清晰；微信并没有像飞书那样在当前官方站点里给出同级独立专页。
  • 但你补充的教程给出了一个实战方案：通过腾讯的 `@tencent-weixin/openclaw-weixin` 插件接入个人微信。
  • 核心动作是安装插件、启用插件、执行 `openclaw channels login --channel openclaw-weixin` 扫码登录，然后重启 gateway。
  • Windows 还提到一个现实坑：`npx` 可能报错，这时可以改用 `openclaw plugins install "@tencent-weixin/openclaw-weixin"` 这条线。

所以课堂里最稳的讲法是：飞书讲官方路径，微信讲“社区/第三方插件实战路径”，不要把两者混成同一种官方支持等级。

WeChat Plugin Flow

如果要讲个人微信接入，可以这样讲清流程

1. 装插件

示例教程使用 `npx -y @tencent-weixin/openclaw-weixin-cli@latest install`，也可以直接用 `openclaw plugins install "@tencent-weixin/openclaw-weixin"`。

2. 开权限

必要时在 `~/.openclaw/openclaw.json` 的 `plugins.allow` 中加入 `openclaw-weixin`。

3. 扫码登录

执行 `openclaw channels login --channel openclaw-weixin`，用微信扫码绑定。

4. 重启并验证

执行 `openclaw gateway restart`，再去后台或聊天窗口验证微信通道是否真正可用。

这条线很适合课堂说明“渠道接入不等于功能魔法”，本质上还是插件、认证、登录和网关重启这些工程动作。

Soul Files

OpenClaw 的“灵魂”不只是一句人设，它是工作区启动文件

  • 官方 Agent Runtime 页面把 `SOUL.md` 视为启动时注入的重要文件之一
  • 同一层还有 `AGENTS.md`、`TOOLS.md`、`IDENTITY.md`、`USER.md` 等文件
  • 这些文件并不是花架子，它们会直接影响 agent 的风格、边界、语气和默认行为
  • 如果工作区是它的“家”，那 `SOUL.md` 就像它的性格与底线说明书

Write Better SOUL

怎么把 `SOUL.md` 写得真正有用？

写边界

什么能做，什么不能做，尤其是外部发送、隐私、公开发言这类高风险动作。

写语气

是简洁、温和、直接，还是更像助教、研究助理、运营搭子，要写清楚。

写偏好

遇到问题是先搜索、先读文件、先提问，还是先给结论，这些都会影响体验。

写禁忌

不要空洞地写“友好”“专业”，要写可执行的原则，比如“外发前先确认”。

Agent Creation

如何根据真实应用创建一个 Agent，而不是只堆一个空壳？

1. 先定任务

先回答它服务什么场景，比如“学习助教”“资料整理员”“群聊答疑机器人”。

2. 再定工作区

给它独立 workspace，把它真正需要读写的资料、模板、记忆放进去。

3. 再写灵魂

用 `SOUL.md` 定语气和边界，用 `AGENTS.md` 定工作规则，用 `USER.md` 写服务对象偏好。

4. 再配路由

如果有多个 agent，用 `openclaw agents add` 和 bindings 把不同渠道、账号或人群路由到不同 agent。

SOUL Example

`SOUL.md` 示例卡片：什么样的灵魂文件才算有用？

好的 `SOUL.md` 有 4 个特点

  • 先写身份：你是谁，服务谁，核心职责是什么
  • 再写边界：什么不能自动做，哪些动作必须先确认
  • 再写风格：回答要简洁还是详细，偏老师、助理还是同伴
  • 再写默认策略：遇到陌生任务先搜索、先读文件还是先提问
你是一个学习助教型 Agent。
你的目标不是替用户完成学习任务，
而是帮助用户拆解任务、整理资料、检查结构、提醒风险。

禁止直接代写需提交的作业。
涉及外发、删除、支付、账号授权前必须先确认。
回答语气保持温和、清晰、直接。
遇到信息不足时，先指出缺口，再建议下一步。

没用的 `SOUL.md` 常常长这样

你是一个专业、友好、强大的 AI 助手，
你无所不能，尽可能帮助用户完成任何任务。
请保持高效、智能、准确。

问题在于：这类写法几乎没有可执行边界。听起来像招聘海报，实际对行为约束帮助很小。

学习助教型 `SOUL.md` 示例

你是一个学习助教型 Agent，服务对象是需要提升阅读、
写作、资料整理和复习效率的学习者。

你优先帮助用户：
1. 整理材料
2. 提炼重点
3. 给出结构化建议
4. 提醒错误和风险

你不直接代写要提交的作业，不替用户做价值判断，
不把不确定的信息说成确定事实。

你的风格像一个耐心但不啰嗦的助教：
先给结论，再给理由，再给下一步建议。

资料整理型 `SOUL.md` 示例

你是一个资料整理型 Agent。
你的核心职责是收集、归类、去重、提纲化和结构化输出。

你默认优先做：
- 分类
- 建清单
- 提炼关键词
- 标出待确认项

遇到来源不明、信息冲突、引用不全时，
必须明确标记“需人工核查”，不能擅自补全。

你的输出应该像一份可继续加工的工作底稿，
而不是像最终定稿。

Create One Agent

真实示例：为“英语学习助教”创建一个 Agent

场景定义

目标不是“什么都做”，而是专注于英语阅读、作文修改、词汇整理和复习提醒。

工作区内容

放入常用阅读材料、作文模板、错题样例、复习任务清单，以及 `SOUL.md / USER.md / AGENTS.md`。

推荐技能

文档整理、网页摘要、OCR、定时提醒、资料分类。不要一上来装十几个花里胡哨的技能。

渠道选择

如果是个人使用，先用 dashboard；如果是课堂或小组协作，再考虑飞书/群聊接入。

Step A

先定边界

它可以整理和提醒，但不能代写作文、不能替用户提交作业。

Step B

再定语气

像助教，不像营销号；像搭子，不像班主任。

Step C

再定默认动作

收到材料时先提炼主旨和关键词；收到作文时先看结构再提建议。

Step D

最后再接渠道

先保证本地或 dashboard 能用，再把它接进飞书或微信，不要一开始就多线开战。

Multi-Agent

官方对“一个 agent”怎么定义？

一个工作区

文件、人格、说明书、技能、记忆，都会围绕这个 workspace 组织。

一个 agentDir

每个 agent 有自己独立的认证、模型注册和状态目录，别混用。

一个会话世界

多 agent 的真正价值是隔离：不同人格、不同账号、不同路由、不同上下文不串味。

官方文档里甚至直接给了 `openclaw agents add work`、`openclaw agents list --bindings` 这类命令，说明多 agent 不是花活，而是系统一级能力。

Deployment Steps

第一次运行时，重点观察什么

第一眼先看它有没有真的“动起来”，别先被礼貌回复骗了。
“配置不是装饰品。你每点开一个开关，其实都在定义它的能力边界和风险边界。”

Troubleshooting

常见翻车位

  • 部署成功了，但不会响应
  • 能聊天，但不会执行动作
  • Skill 装了，但不会正确调用
  • 上下文越来越乱
  • 平台额度、会员、模型权限没跟上

Boundary Warning

能执行，不代表永远执行得对

  • 不要轻易给高权限
  • 不要在重要环境里乱试
  • 不要把模糊目标直接丢给强执行 Agent
  • 不要忘了会员、额度、平台限制也属于现实边界
  • 不要把“一键部署”理解成“没有学习成本”

Lecture 03

让 Agent 帮助学习，顺便学会选工具

建议时长：120 分钟 目标：学会用，也学会停

英语阅读

适合：主旨提炼、词汇表、复习问题。
不适合：把理解过程整包外包。

作文修改

适合：结构检查、逻辑提醒、语法建议。
不适合：整篇代写直接提交。

错题归类

适合：按错误原因分类、找规律。
不适合：把“为什么错”也一并不思考。

资料整理

适合：收集、去重、分层次。
不适合：把可信度判断彻底交出去。

Case Archive

真实案例库：Agent 到底能把哪些事串成工作流？

案例 1：监控网页变化并主动提醒

文档里提到过一个很适合讲清 Agent 味道的例子：不是只让 AI 回答“最新动态是什么”，而是让它定期去看、去截图、去比较、去提醒。

  • 适合讲“常驻待命”和“触发式任务”
  • 适合说明为什么浏览器能力和截图分析有价值
  • 也适合提醒：这类任务明确、重复、可检查，所以很适合 Agent

案例 2：把 OpenClaw 当成第二大脑入口

不是让它替你思考，而是让它记住你的笔记、想法、任务和线索，需要时帮你回收、整理、再展示。

  • 适合讲“记忆”和“上下文”的价值
  • 适合展示为什么 Agent 比普通聊天框更像长期协作对象
  • 也适合提醒：保存了很多，不等于理解了很多

案例 3：发票 OCR + 飞书多维表格

这是一个非常适合解释“Skill 即插即用”和“任务编排”的案例：上传发票，OCR 解析，写入表格，形成自动化闭环。

  • 适合讲 Skill、API、平台集成三者如何组合
  • 适合说明 Agent 的价值不是单点能力，而是串联流程
  • 也适合提醒：现实里仍然需要授权、字段校对和规则修正

案例 4：内容生产流水线

比如下载视频、提取信息、生成摘要、产出图文草稿、存到平台文档，这种“多个环节连续执行”的工作特别有 Agent 味。

  • 适合讲“为什么它看起来像在干活”
  • 适合说明平台型方案为什么对小白更有吸引力
  • 也适合强调：越自动，越要注意版权、真实性和审校

案例 5：知识管理和资料沉淀

把资料分类、打标签、转成结构化内容，甚至接入知识库，这类任务很适合 Agent 先帮你打底。

  • 适合讲学习场景与工作场景是相通的
  • 适合连接英语学习、错题整理、资料库建设
  • 也适合提醒：知识库丰富，不代表人就自动长知识

Good Tasks

更适合 Agent 的任务画像

  • 步骤明确
  • 可以机械执行
  • 结果可回看、可检查
  • 重复率高，手工做很烦
  • 适合被技能或工具串起来形成闭环

Bad Tasks

不该盲目交给 Agent 的任务画像

  • 需要连续价值判断
  • 目标在过程中频繁变化
  • 高风险、不可逆
  • 本质是训练你的理解、判断、表达能力
  • 结果一旦错了，代价非常高
“工具的上限决定它能帮你走多快，你的判断力决定它会不会把你带偏。”

Final Shrine

这门课希望带走的 5 个判断

01

Agent 不只是聊天 AI 的升级版，它代表的是从回答问题到执行任务的变化。

02

OpenClaw 一类工具之所以吸引人，不只是因为新，而是因为它更像真正的数字助理。

03

便利、成本、隐私和风险，永远要一起看，不能只看“它看起来好厉害”。

04

真正重要的不是记住多少工具名字，而是学会判断哪一类工具适合你的目标。

05

AI 最好的位置不是替你学习，而是把那些机械、重复、可拆解的活先接过去。
